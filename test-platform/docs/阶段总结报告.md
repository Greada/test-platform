## 四、技术决策回顾

### 1. JSON 匹配策略

```
expected_result 为合法 JSON
  → JSON 字段子集匹配（递归嵌套，忽略多余字段）
  → 实际响应中必须包含预期中的所有字段且值一致
expected_result 非 JSON
  → contains 文本降级匹配
```

**理由**：真实 API 响应通常包含大量额外字段（时间戳、ID 等），全等匹配过于严格导致全部 FAIL。字段子集匹配兼顾准确性和灵活性。

### 2. 执行记录快照

`execution_record` 新增 `test_no` 和 `case_name` 字段，在批量执行时从 `test_case` 表快照写入。

**理由**：测试用例可能被修改或删除，历史执行报告应保持当时的准确性。如果仅通过 `test_case_id` 关联，报告会随用例修改而变化。

### 3. 无 execution_report_detail 表

不建独立的明细表，而是通过 `execution_record.report_id` 字段关联查询。

**理由**：减少表数量和同步复杂度，`execution_record` 已有 `report_id` 字段足以支持按报告查明细。

### 4. HttpResult 包装类

`HttpExecutor.execute()` 返回 `HttpResult { body, duration, statusCode }` 而非原生的 `String`。

**理由**：执行耗时和状态码是报告统计的必要数据，多值返回需要包装类。

### 5. 批量添加接口

`POST /api/test-suites/{id}/cases/batch` 接收 `{caseIds: [1,2,3]}`。

**理由**：前端频繁逐条调用添加接口会产生 N 次 HTTP 请求，批量接口一次完成，提升体验和性能。

### 6. 加载对话框

批量执行时显示非关闭式弹窗（loading + 文案）。

**理由**：套件执行可能涉及多个用例，逐个 HTTP 调用有明显等待时间。加载弹窗防止用户误操作（重复点击、刷新页面、跳转）。

### 7. OpenAPI 导入 — 本地生成 vs AI 生成

导入支持 `useAi` 开关：默认用 SchemaToJsonGenerator 本地从 JSON Schema 生成模板 JSON（免费、毫秒级），可选切换 AI 生成（需 AGNES_API_KEY、耗时较长）。

**理由**：AI 生成预期结果成本高、速度慢，本地模板生成满足大部分场景，用户按需选择。

### 8. JWT 密钥推导

使用 SHA-256 从配置的 secret 固定推导 32 字节 HMAC 密钥，确保任意长度的 secret 都满足 HS256 要求。

**理由**：jjwt 0.12.x 要求 256 位密钥，直接使用短 secret 会报警告或选择弱算法。

### Docker 部署教训

| 教训 | 具体问题 | 改进措施 |
|---|---|---|
| Nginx 容器启动顺序 | 启动时 backend 未就绪，"host not found in upstream backend" | 用变量 `$backend_upstream` + `resolver 127.0.0.11` 运行时解析 |

### CI/CD 部署教训

| 教训 | 具体问题 | 改进措施 |
|---|---|---|
