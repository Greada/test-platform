## 二、技术决策回顾

### 1. JSON 匹配策略

```
expected_result 为合法 JSON
  → JSON 字段子集匹配（递归嵌套，忽略多余字段）
  → 实际响应中必须包含预期中的所有字段且值一致
expected_result 非 JSON
  → contains 文本降级匹配
```

**理由**：真实 API 响应通常包含大量额外字段（时间戳、ID 等），全等匹配过于严格导致全部 FAIL。字段子集匹配兼顾准确性和灵活性。

### 2. 执行记录快照

`execution_record` 新增 `test_no` 和 `case_name` 字段，在批量执行时从 `test_case` 表快照写入。

**理由**：测试用例可能被修改或删除，历史执行报告应保持当时的准确性。如果仅通过 `test_case_id` 关联，报告会随用例修改而变化。

### 3. 无 execution_report_detail 表

不建独立的明细表，而是通过 `execution_record.report_id` 字段关联查询。

**理由**：减少表数量和同步复杂度，`execution_record` 已有 `report_id` 字段足以支持按报告查明细。

### 4. HttpResult 包装类

`HttpExecutor.execute()` 返回 `HttpResult { body, duration, statusCode }` 而非原生的 `String`。

**理由**：执行耗时和状态码是报告统计的必要数据，多值返回需要包装类。

### 5. 批量添加接口

`POST /api/test-suites/{id}/cases/batch` 接收 `{caseIds: [1,2,3]}`。

**理由**：前端频繁逐条调用添加接口会产生 N 次 HTTP 请求，批量接口一次完成，提升体验和性能。

### 6. 加载对话框

批量执行时显示非关闭式弹窗（loading + 文案）。

**理由**：套件执行可能涉及多个用例，逐个 HTTP 调用有明显等待时间。加载弹窗防止用户误操作（重复点击、刷新页面、跳转）。

## 三、经验教训

### V1 阶段教训

| 教训 | 具体问题 | 改进措施 |
|---|---|---|
| 预期结果为空无校验 | 直接存入空字符串，执行时异常 | DB + 后端 + 前端三层 NOT NULL 校验 |
| 匹配逻辑过于简单 | 全等匹配全部 FAIL | 改为字段子集递归匹配，非 JSON 文本降级 |
| SecurityConfig 路径错误 | `"*/**"` 语法错误导致 403 | 了解 Spring Security ant 匹配规则 |
| 缺少 HttpExecutor 测试 | 不解析 headers/body | 后续版本应加入单元测试 |

### V2.1 阶段教训

| 教训 | 具体问题 | 改进措施 |
|---|---|---|
| 空壳 Controller | 先写接口再实现导致遗漏 | AI 生成时一次输出完整实现 |
| 字段名不统一 | Java 字段 `executionTime` vs DB 字段 `execute_time` | 统一命名规范，MyBatis-Plus 自动驼峰映射 |

## 八、V2.2 阶段

> 阶段：V2.2 完成
> 日期：2026-06-07

### 1. 阶段概述

V2.2 聚焦**调试体验增强**和**质量修复**两大方向。新增 JSON Diff 可视化对比、自动修复建议、错误模式聚合三个核心功能。同时修复了批执行 ERROR 时 responseDetail 为 null 的 bug，并对 ExecutionServiceImpl 进行了结构化重构。

### 2. 新增功能

| 功能 | 说明 | 收益 |
|---|---|---|
| JSON Diff 可视化 | 双栏对比预期/实际 JSON，差异字段逐行展示（MISMATCH/MISSING/EXTRA） | 一眼定位执行失败原因 |
| 自动修复建议 | 分析差异后自动生成修复方案（格式化的 actualResult），一键应用 | 减少手动复制粘贴 |
| 错误模式聚合 | 按 URL+Method 分组统计通过率，标出最差端点 | 快速发现稳定性短板 |

### 3. 修复记录

| # | 问题 | 修复 |
|---|---|---|
| 1 | 批执行 HTTP 4xx/5xx 时 responseDetail 为 null | HttpExecutor 增加 HttpStatusCodeException 捕获 |
| 2 | ERROR 路径 missing executeDuration | ExecutionServiceImpl 错误路径显式 setExecuteDuration(0L) |
| 3 | execute() 方法过长（~65 行） | 抽取 5 个辅助方法（buildBaseRecord/buildRequestDetailJson/buildErrorResult/buildErrorDetailJson/resolveStatus） |
| 4 | ObjectMapper 重复 new（3 次） | 改为构造注入，全局复用 |
| 5 | 嵌套 JSON 在 diff 中显示为 [object Object] | 前端 deepUnescape() 递归展开嵌套字符串 |

### 4. 技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| Diff 计算方式 | 实时计算（不加新表） | actualResult 从 execution_record，expectedResult 从 test_case 获取，无数据冗余 |
| 错误模式聚合 | 纯 Java Stream + LinkedHashMap（不用 SQL GROUP BY） | 报告级数据量小，内存处理无压力，避免 XML mapper 文件 |
| 修复建议存储 | localStorage（不用 API PATCH） | 简单，用户确认后再手动保存 |
| 前端组件化 | JsonDiffViewer + ErrorPatternCard 独立组件 | 可复用、可单独测试 |
